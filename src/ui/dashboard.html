conhecimento interno do Orion
🔭 Visão

Orion é o orquestrador autônomo pessoal do Danilo. Um único cérebro que:

  • Recebe tarefas por qualquer canal — WhatsApp, plugin Claude Code, chat próprio
  • Tem memória unificada — fala no WhatsApp, lembra no plugin. Zero diferença de canal.
  • Delega código e infra via Delegate — sub-agentes que aparecem como sessões no plugin
  • Cria cron jobs por linguagem natural
  • Aprende continuamente: raw → episodic → semantic → skill → vault

🧠 O que é o sistema de memória do Orion

O Orion não tem uma memória — ele tem seis pipelines de memória paralelos que operam em camadas de tempo diferentes:

  • Instantâneo (por turno): extração por regex + Haiku + background review → banco SQLite
  • Curto prazo (30 min): Phase 2 — deduplicação BM25, decay de confiança, promoção raw→episodic
  • Médio prazo (6h): Phase 3 — merge semântico Haiku, contradições, escrita no vault
  • Diário (meia-noite): Phase 4 — arquivamento de memórias fracas, relatório WhatsApp
  • Semanal (domingos 9h): Phase 5 — self-audit Haiku: avalia qualidade das memórias mais usadas
  • Mensal (1º do mês): Phase 6 — ground-truth: verifica se fatos antigos ainda são verdade

Cada memória tem: type · category · confidence · source_channel · source_session_id · source_tool · helpful_votes · unhelpful_votes · access_count · tags · vetor embedding 384 dims

⚡ Pipeline por turno — o que acontece após cada resposta

Tudo abaixo roda em setImmediate (background, sem bloquear a resposta).

extractFacts()
extraction.js
Regex rápida, zero LLM. Detecta padrões como "meu projeto é X", "uso Y", nomes de pessoas, decisões explícitas. Salva como type: raw, confidence: 0.5. Roda em <1ms.
extractEntities()
extraction.js
Knowledge graph leve. Extrai entidades (pessoas, projetos, ferramentas) e relações ("usa", "trabalha em", "prefere") → tabelas entities e relations. Também zero LLM.
extractWithHaiku()
memory/index.js · Haiku
Extração semântica real. Haiku analisa a troca completa e retorna 0–3 fatos no formato CATEGORIA|TAGS|fato. Só roda se ambas as mensagens têm >20 chars. Salva como type: episodic, confidence: 0.65, sourceTool: haiku-extraction.
backgroundReview()
background-reviewer.js · Haiku
Segundo olhar pós-turno. Haiku revisa a troca buscando o que a extração padrão pode ter perdido: fatos implícitos, candidatos a skill, hints de contradição. Retorna JSON { new_facts, skill_candidate, contradiction_hint }. Salva com sourceTool: background-review, confidence: 0.4 (mais conservador).
🎯 Como memórias são ranqueadas — fórmula de score

Quando o Orion busca memórias relevantes para uma mensagem, cada candidata recebe um score composto:

score = 0.40 × BM25_norm
      + 0.30 × vector_similarity
      + 0.15 × confidence × decay(category, age)
      + 0.15 × recency_score(created_at)
      + feedback_boost(helpful_votes, unhelpful_votes)
BM25 (0.40)
Full-text search via FTS5. Normalizado: |score| / (|score| + 1). Melhor para fatos explícitos mencionados na mensagem.
Vector similarity (0.30)
Cosseno no espaço all-MiniLM-L6-v2 (384 dims, local). Melhor para conceitos relacionados mesmo sem palavras idênticas.
Decay adaptativo (0.15)
conf × e^(−rate × age_days)
Taxas por categoria:
decision 0.015 · tool 0.008 · general 0.005 · project 0.003 · user_pref 0.002 · person 0.001
Recência (0.15)
e^(−λ × age_seconds). Decai ao longo de dias mas nunca zera. Garante que memórias recentes têm leve preferência.
Feedback boost (±0.08)
(hv − uv×2) / (hv + uv×2 + 1) × 0.08
Assimétrico: voto negativo pesa 2× mais que positivo. Memórias com feedback ruim ficam para baixo. Memória com 1 positivo: +0.027. Com 1 negativo: −0.053. Sem votos: neutro.

🔍 Rastreabilidade — de onde veio cada memória

Cada memória tem 3 colunas de provenance além do campo source padrão:

CampoO que armazenaExemplos
source_channelCanal de origem da conversawhatsapp · plugin · chat_ui
source_session_idUUID da sessão Claude Code ou WhatsApp4e352b4b-8a74-...
source_toolPipeline específico que criou a memóriahaiku-extraction · background-review · pre-compress · delegate · phase3-merge · mcp-save
🆕 4 melhorias recentes — paridade com Hermes + segurança
A on_pre_compress — extrai fatos antes de comprimir memory/compressor.js

O problema: quando uma sessão com >40 mensagens era comprimida, as mensagens descartadas iam para o lixo sem extração de fatos. Decisões tomadas, preferências mencionadas, projetos discutidos — tudo perdido.
A solução: antes do soft-delete, um Haiku analisa o transcript completo das mensagens a serem descartadas e extrai até 5 fatos no formato CATEGORIA|TAGS|fato. Salvos com sourceTool: pre-compress. Roda em setImmediate, não atrasa a resposta.

B on_delegation — captura resultado do subagente agent/delegate.js

O problema: quando o Orion delegava uma tarefa (ex: "analisa o Brandspace"), o subagente executava, produzia output — e esse output nunca entrava na memória. Na próxima sessão, o Orion não lembrava o que tinha feito.
A solução: após cada delegação bem-sucedida, salva automaticamente: "Tarefa delegada (role): [goal] → [snippet do resultado]". Tags incluem delegate, o role e o projeto. Categoria decision.

C Feedback no score de retrieval memory/index.js

O problema: os campos helpful_votes e unhelpful_votes existiam no banco mas eram ignorados no ranking. Feedback do usuário não tinha impacto nenhum.
A solução: o score agora inclui feedback_boost = (hv − uv×2) / (hv + uv×2 + 1) × 0.08. Assimétrico: voto negativo penaliza 2× mais que positivo recompensa. Memórias marcadas como inúteis afundam no ranking progressivamente.

D Threat detection — anti injeção de prompt memory/threat-check.js

O problema: qualquer mensagem enviada pelo WhatsApp podia conter injeção de prompt que seria persistida no banco e servida como contexto em todas as respostas futuras.
A solução: antes de qualquer saveMemory(), o conteúdo passa por threatCheck() com 16 regex patterns (ignore instructions, you are now, reveal system prompt, eval/exec, etc). Se match: salva com confidence: 0.01 e tags ["blocked","threat"] — preservado para auditoria, mas nunca surfaceado no retrieval.

🔮 HRR Composer — multi-entity queries algébricas

Quando você quer memórias que envolvem múltiplas entidades ao mesmo tempo (ex: "o que sei sobre Danilo E Orion"), o BM25 padrão faz duas buscas separadas. O HRR Composer combina as entidades em um único vetor de consulta via álgebra de espaço de fases:

normalize(v) → ||v|| = 1, Float32Array
bindVectors(a, b) → produto Hadamard + normalize
superpose([v1, v2, ...]) → soma ponderada + normalize

retrieveByComposedEntities(["Danilo", "Orion"], limit=10)
  → embedding("Danilo") + embedding("Orion")
  → superpose → KNN search no vec_memories

Endpoint: GET /api/memories/entities-composed?names=A,B&limit=10
